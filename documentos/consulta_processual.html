Integração - Consulta Processos via webservices

1. Introdução

A operação consultarProcesso visa a permitir consulta a um determinado processo judicial, retornando seus dados, documentos e movimentos.

Informados os dados de entrada, a operação efetua a consulta na base de dados do Tribunal e retorna um indicador de sucesso, uma mensagem relativa à realização da operação e os dados encontrados de acordo com os parâmetros registrados. Caso ocorra algum problema ao efetuar a consulta, a operação retorna um indicador de que não foi bem sucedida e uma mensagem de acordo com o erro correspondente.

Não é necessário que o consultante seja representante de qualquer das partes componentes do processo.

A identificação do requerente somente será necessária quando inexistente a autenticação e autorização pelo certificado cliente do serviço.

A implementação deverá assegurar que o processo somente seja retornado se o nível de sigilo interno permitir a consulta pelo requerente.

A operação retornará objeto do tipo tipoConsultarProcessoResposta, que contém, além dos dados básicos, um objeto do tipo tipoProcessoJudicial, definido pelo esquema http://www.cnj.jus.br/intercomunicacao-2.2.2, caso o processo exista e possa ser acessado pelo consultante.

Ação SOAP: http://www.cnj.jus.br/servico-intercomunicacao-2.2.2/consultarProcesso

 

2. Formas de autenticação

Há duas formas possíveis de autenticação no MNI 2.2 do Eproc: autenticação pessoal e autenticação institucional.

 

2.1. Autenticação pessoal

No caso da autenticação pessoal, o elemento idConsultante deve ser preenchido com o CPF ou a sigla do usuário. A senhaConsultante é senha pessoal do usuário, a mesma utilizada pela interface do Eproc.

Sabemos que no Eproc, é possível uma mesma pessoa possuir mais de um perfil. Por exemplo, um procurador pode possuir perfil específico para sua atuação como procurador-chefe (perfil que é incluído na citação eletrônica como representante da entidade) e outro perfil para sua atuação como procurador de fato (processos em que o procurador está atuando). Neste caso, não será possível utilizar a autenticação pessoal utilizando o CPF da pessoa, pois o Eproc localizará mais de um perfil para o mesmo CPF. Ao invés disso, os perfis do procurador devem possuir siglas diferentes e o acesso ao serviço deve ser feito por sigla (no caso de autenticação pessoal é claro).

Com a autenticação pessoal é possível consultar qualquer processo que o usuário teria acesso na interface web do Eproc. Desta forma, ainda que um processo tenha o nível máximo de sigilo (nível 5), desde que o procurador tenha permissão de acesso a este processo, a consulta deste pelo serviço MNI é permitida.

 

2.2. Autenticação institucional

Na autenticação institucional, o elemento idConsultante deve ser preenchido com a identificação institucional da entidade. Em geral, é utilizado o CNPJ desta como idConsultante. A senhaConsultante, neste caso, é uma senha exclusiva para acesso ao serviço. Esta senha é fornecida pela equipe de TI do Tribunal para o acesso da entidade.

No caso de utilização de autenticação institucional para chamada à operação consultarProcesso, é permitida a consulta de processos com nível de sigilo 0, 1 e 2. Processos com sigilo 3, 4 ou 5 não são retornados na operação consultarProcesso quando utilizada autenticação institucional.

 

3. Envio da solicitação

 

3.1. Consulta dados básicos do processo, movimentações e metadados dos documentos

A consulta processual do MNI, a chamada operação 3 do MNI, é feita com base no número único do processo.

Por meio da consulta dos avisos pendentes, obtém-se os números dos processos em que o consultante tem atuação. De posse de um número de processo, o consultante pode chamar a operação consultarProcesso, a fim de obter as informações do processo, bem como os documentos que foram produzidos no consultado.

O elemento incluirDocumentos informado como true indica que a identificação dos documentos deve ser retornada, mas não o seu conteúdo. Para o retorno do conteúdo dos documentos, deve ser realizada uma segunda consulta na qual sejam passados os identificadores dos documentos.

Segue exemplo de consulta processual que retorna os dados básicos do processo, movimentos e metadados dos documentos.

<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/"
   xmlns:ser="http://www.cnj.jus.br/servico-intercomunicacao-2.2.2/"
   xmlns:tip="http://www.cnj.jus.br/tipos-servico-intercomunicacao-2.2.2">
   <soapenv:Header/>
   <soapenv:Body>
      <ser:consultarProcesso>
         <tip:idConsultante>ID_CONSULTANTE</tip:idConsultante>
         <tip:senhaConsultante>SENHA_CONSULTANTE</tip:senhaConsultante>
         <tip:numeroProcesso>50000813720188210044</tip:numeroProcesso>
         <!-- Opcional: caso utilizada, retorna apenas os movimentos posteriores à data/hora informada -->
         <!--<tip:dataReferencia>20180423092500</tip:dataReferencia>-->
         <!-- Indica se devem ser retornados os eventos do processo: true/false -->
         <tip:movimentos>true</tip:movimentos>
         <!-- Indica se devem ser retornados os dados básicos do processo: true/false -->
         <tip:incluirCabecalho>true</tip:incluirCabecalho>
         <!-- Indica se devem ser retornados os identificadores (idDocumento) dos documentos: true/false -->
         <tip:incluirDocumentos>true</tip:incluirDocumentos>
      </ser:consultarProcesso>
   </soapenv:Body>
</soapenv:Envelope>

3.2. Passo 2  Consulta conteúdo dos documentos

Uma vez executada a consulta descrita na seção 3.1, a aplicação cliente do serviço MNI terá acesso aos identificadores dos documentos (idDocumento). Com esses identificadores, é possível consultar o conteúdo dos documentos, conforme ilustrado a seguir. Nesta consulta é possível informar um ou mais documentos.

Sugere-se que seja utilizado o hash do documento para verificar se este já não se encontra na base de dados do consultante, desta forma efetuando o download apenas dos documentos produzidos no processo consultado.

<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/"
   xmlns:ser="http://www.cnj.jus.br/servico-intercomunicacao-2.2.2/"
   xmlns:tip="http://www.cnj.jus.br/tipos-servico-intercomunicacao-2.2.2">
   <soapenv:Header/>
   <soapenv:Body>
      <ser:consultarProcesso>
         <tip:idConsultante>ID_CONSULTANTE</tip:idConsultante>
         <tip:senhaConsultante>SENHA_CONSULTANTE</tip:senhaConsultante>
         <tip:numeroProcesso>50000813720188210044</tip:numeroProcesso>
         <tip:movimentos>false</tip:movimentos>
         <tip:incluirCabecalho>false</tip:incluirCabecalho>
         <!-- Indica quais documentos devem ser retornados (conteúdo) -->
         <tip:documento>11524485384634000633983466690</tip:documento>
         <!-- Pode ser informado mais de um documento por requisição -->
         <tip:documento>11524485384634000633934544550</tip:documento>
      </ser:consultarProcesso>
   </soapenv:Body>
</soapenv:Envelope>

Pode-se verificar no exemplo acima que os elementos incluirCabecalho e movimento foram indicados como false. O motivo é que se o objetivo da consulta é apenas retornar os binários dos documentos, é possível dispensar o processamento para exibição dos dados básicos e movimentos.

É possível ainda solicitar o retorno de todos os documentos do processo sem a necessidade de informar os identificados destes. Isto é possível com o uso de * em vez do identificador do documento, conforme exemplo a seguir.

<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/"
   xmlns:ser="http://www.cnj.jus.br/servico-intercomunicacao-2.2.2/"
   xmlns:tip="http://www.cnj.jus.br/tipos-servico-intercomunicacao-2.2.2">
   <soapenv:Header/>
   <soapenv:Body>
      <ser:consultarProcesso>
         <tip:idConsultante>ID_CONSULTANTE</tip:idConsultante>
         <tip:senhaConsultante>SENHA_CONSULTANTE</tip:senhaConsultante>
         <tip:numeroProcesso>50000813720188210044</tip:numeroProcesso>
         <tip:movimentos>false</tip:movimentos>
         <tip:incluirCabecalho>false</tip:incluirCabecalho>
         <!-- Todos documentos devem ser retornados (conteúdo) -->
         <tip:documento>*</tip:documento>
      </ser:consultarProcesso>
   </soapenv:Body>
</soapenv:Envelope>

4. Prioridades

Um dos itens retornados na consulta processual do MNI é o elemento prioridade. Segundo o MNI, prioridade é o: Elemento destinado a permitir a identificação da existência de propriedades processuais não óbvias, ou seja, aquelas que não são resultado direto da identificação da classe processual (ex.: habeas corpus ou mandado de segurança).

Como o elemento é definido como um texto livre, esta seção visa listar o domínio de valores possíveis para este elemento na implementação da operação consultarProcesso existente no Eproc. Segue a lista de valores possíveis para este elemento.

  • Admitida execução
  • Agravo Retido
  • Antecipação de Tutela - Deferida
  • Antecipação de Tutela - Parcialmente Deferida
  • Antecipação de Tutela - Requerida
  • Autor manifesta desinteresse na conciliação
  • Criança e Adolescente
  • Criminal
  • Doença Grave
  • Efeito Suspensivo
  • Idoso - maior de 80 anos
  • Idoso
  • Justiça Gratuita - Deferida
  • Justiça Gratuita - Parcialmente Deferida
  • Justiça Gratuita - Requerida
  • Penhora no rosto dos autos
  • Penhora/apreensão de bens
  • Pessoa com deficiência
  • Petição Urgente
  • Prioridade Atendimento
  • Reconvenção
  • Renúncia excedente 60 salários
  • Réu Preso
  • Réu sob Monitoramento
  • Vista Ministério Público

 

5. Movimentos e complementos

Esta seção descreve o conteúdo dos complementos de movimentação retornados na consulta processual via MNI.

A descrição do evento é retornada sempre no atributo descricao do elemento movimentoLocal.

As duas primeiras posições do complemento estarão sempre preenchidas. Segue a descrição de cada posição.

Posição 1: Retorna o complemento do evento. Caso o evento não possua complemento, ainda assim o elemento será retornado, mesmo que vazio (exemplo <ns2:complemento></ns2:complemento>).

Posição 2: Retorna o nome do usuário que efetuou a movimentação. O formato a ser retornado segue o padrão Movimentado por: [Nome da Pessoa] - [Tipo de Usuário]. A seguir, há mais duas possíveis posições.

Em eventos de decisão (despacho, decisão, sentença, etc.), retorna o magistrado responsável pela movimentação no formato Responsável: [Nome do magistrado].

Prazo: em eventos de citação/intimação, retorna detalhes do prazo no seguinte formato Prazo: [número de dias do prazo] dias - Status: [status do prazo] - Parte: [Nome da Parte] ([Tipo da parte]).

Exemplos do formato retornado:

  • Exemplo com magistrado:
<ns2:movimento dataHora="20171122025741" identificadorMovimento="54">
   <ns2:complemento>Requisição no. 123456789 processada no TRF4 com o no. 50230000020174049666 (DORVALINO DA ROSA)</ns2:complemento>
   <ns2:complemento>Movimentado por: GUILHERME PINHO MACHADO - MAGISTRADO</ns2:complemento>
   <ns2:complemento>Responsável: GUILHERME PINHO MACHADO</ns2:complemento>
   <ns2:movimentoLocal codigoMovimento="350203" codigoPaiNacional="0" descricao="Requisição de Pagamento - Pequeno Valor - Remetida"/></p>
   <ns2:idDocumentoVinculado>41511321401000121104086460000</ns2:idDocumentoVinculado>
</ns2:movimento>
  • Exemplo com prazo:
<ns2:movimento dataHora="20180108145740" identificadorMovimento="57">
   <ns2:complemento>Refer. ao Evento: 56</ns2:complemento>
   <ns2:complemento>Movimentado por: ANA CRISTINA PINHEIRO SILVEIRA CARDOSO - SUPERVISOR</ns2:complemento>
   <ns2:complemento>Prazo:10 dias - Status: FECHADO - Parte: DORVALINO DA ROSA (AUTOR)</ns2:complemento>
   <ns2:movimentoLocal codigoMovimento="17020101" codigoPaiNacional="0" descricao="Intimação Eletrônica - Expedida/Certificada"/>
</ns2:movimento>

 

6. Parâmetros adicionais

Além dos parâmetros padrão MNI, o Eproc também implementa parâmetros adicionais para a operação consultarProcesso.

 

6.1. Chave de consulta

Processos sigilosos podem ser consultados contanto que seja fornecida a chave de consulta usando o parâmetro adicional chave. Por exemplo:

<tip:outroParametro nome="chave" valor="694939105218"/>

 

6.2. Descrições

Por padrão, a resposta do webservice não descreve a classe e o assunto do processo (apenas retorna os códigos), assim como não descreve os pólos ativo e passivo. Para obter essas descrições, o parâmetro adicional incluirDescricoes deve ser definido como true:

<tip:outroParametro nome="incluirDescricoes" valor="true"/>

Exemplo de resposta incluindo descrições:

<SOAP-ENV:Envelope ...>
   <SOAP-ENV:Body>
      <ns3:consultarProcessoResposta>
         <ns1:sucesso>true</ns1:sucesso>
         <ns1:mensagem>Consulta realizada com sucesso</ns1:mensagem>
         <ns1:processo>
            <ns2:dadosBasicos ...>
               ...
               <ns2:outroParametro nome="desClasse" valor="Agravo de Instrumento"/>
               <ns2:outroParametro nome="desAssunto" valor="Dívida Ativa não-tributária, DIREITO ADMINISTRATIVO E OUTRAS MATÉRIAS DE DIREITO PÚBLICO"/>
               <ns2:outroParametro nome="desTipoParteAT" valor="AGRAVANTE"/>
               <ns2:outroParametro nome="desTipoPartePA" valor="AGRAVADO"/>
               ...
            </ns2:dadosBasicos>
         </ns1:processo>
      </ns3:consultarProcessoResposta>
   </SOAP-ENV:Body>
</SOAP-ENV:Envelope>

 

6.3. Documentos e movimentos de outro grau

Por padrão, documentos e movimentos de outro grau não são retornados pelo webservice. Para obtê-los, definir o parâmetro adicional movimentosOutroGrau como true:

<tip:movimentos>true</tip:movimentos>
<tip:outroParametro nome="movimentosOutroGrau" valor="true"/>

O parâmetro adicional movimentosOutroGrau deve ser usado junto com os parâmetros incluirDocumentos e/ou movimentos.

 

6.4. Endereços das partes

Por padrão, os endereços das partes são retornados pelo webservice. Para suprimi-los, definir o parâmetro adicional incluirEnderecos como false:

<tip:outroParametro nome="incluirEnderecos" valor="false"/>

 

6.5. Hash das movimentações

Para retornar o hash SHA-1 do conjunto de identificadores das movimentações ocorridas no processo judicial, definir o parâmetro incluirHashMovimentacoes como true:

<tip:outroParametro nome="incluirHashMovimentacoes" valor="true"/>

Exemplo de resposta:

<SOAP-ENV:Envelope ...>
   <SOAP-ENV:Body>
      <ns3:consultarProcessoResposta>
         <ns1:sucesso>true</ns1:sucesso>
         <ns1:mensagem>Consulta realizada com sucesso</ns1:mensagem>
         <ns1:processo>
            <ns2:dadosBasicos ... >
               ...
               <ns2:outroParametro nome="hashMovimentacoes" valor="77de68daecd823babbb58edb1c8e14d7106e83bb"/>
               ...
            </ns2:dadosBasicos>
         </ns1:processo>
      </ns3:consultarProcessoResposta>
   </SOAP-ENV:Body>
</SOAP-ENV:Envelope>

 

6.6. Limitar partes

Para processos com muitas partes, é possível limitar a no máximo 3 (três) as partes na capa do processo definindo o parâmetro limitarPartes como true:

<tip:outroParametro nome="limitarPartes" valor="true"/>

 

6.7. Paginação de movimentos

Opcionalmente é possível paginar os movimentos processuais usando os parâmetros limitarMovimentos e paginaMovimentosAtual. Quando usados, ambos devem ser informados. Os movimentos são ordenados do último (mais recente) ao primeiro (mais antigo) e a contagem das páginas começa em 0 (zero).

Por exemplo, em um processo com 34 movimentos, para retornar do movimento 24 ao 15, a requisição deve conter:

<tip:outroParametro nome="limitarMovimentos" valor="10"/>
<tip:outroParametro nome="paginaMovimentosAtual" valor="1"/>

Quando os parâmetros limitarMovimentos e paginaMovimentosAtual são informados, o sistema assume que o parâmetro movimentos recebe o valor true, dispensando a informação desse parâmetro.
